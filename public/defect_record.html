BUG-01 关联用例：TC11（库存调整归零）
缺陷现象库存调整接口返回 HTTP 500 内部错误，报错信息：SqliteError: no such column: now
原因分析server.js 中 datetime() 函数使用了双引号：datetime("now","localtime")。
SQLite 将双引号视为标识符（列名）而非字符串字面量，导致 "now" 被当作列名解析，触发 "no such column" 错误。
🔴 修复前（错误代码）：
// server.js 第 329 行
  VALUES (?, ?, ?, ?, datetime("now","localtime"))

// server.js 第 341 行
  created_at = datetime("now","localtime")
🟢 修复后（正确代码）：
// server.js 第 329 行
  VALUES (?, ?, ?, ?, datetime('now','localtime'))

// server.js 第 341 行
  created_at = datetime('now','localtime')
✅ 修复结果：将 datetime("now","localtime") 改为 datetime('now','localtime')（双引号 → 单引号）。修复后重新执行测试，TC11 通过，库存归零功能正常。
BUG-02 关联用例：TC09（空密码登录）
缺陷现象空密码登录测试预期返回 HTTP 401，但实际返回 HTTP 400，导致测试用例 AssertionError 失败。
原因分析测试用例预期值设定错误。server.js 第 35 行将空用户名/密码归类为"缺少必填字段"的 400 Bad Request，而非 401 Unauthorized。
实际上 400 更符合 RESTful 规范：客户端请求格式有误，而非认证凭据无效。
🔴 修复前（测试用例错误预期）：
// run_tests.py — TC09 原始代码
test("TC09", "边界", lambda: (
    (r := requests.post(f"{BASE}/api/auth/login",
        json={"username": "admin", "password": ""})),
    assert_status(r, 401),  ❌ 预期 401，实际 400 → 断言失败
    r.json()["code"] != 0 or ...
))
🟢 修复后（对齐实际接口行为）：
// run_tests.py — TC09 修正后代码
test("TC09", "边界", "空密码登录（返回 400 Bad Request）", lambda: (
    (r := requests.post(f"{BASE}/api/auth/login",
        json={"username": "admin", "password": ""})),
    assert_status(r, 400),  ✅ 修正为 400，与实际接口一致
    r.json()["code"] != 0 or ...
))
💡 结论：此条为测试用例编写失误，非服务端代码缺陷。服务端的 400 Bad Request（缺少必填字段）处理方式符合 RESTful API 设计规范。
✅ 修复结果：将 TC09 的 HTTP 预期状态码从 401 修正为 400。修正后 TC09 通过，与接口实际行为对齐。
